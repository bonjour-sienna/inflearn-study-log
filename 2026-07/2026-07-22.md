## 1. 컨텍스트 · 토큰 관리

### 1-1. 반복 작업은 스크립트로 전환

포맷 변경처럼 반복되는 일은 매번 대화로 시키지 말고, 한 번 스크립트를 작성시킨 뒤
다음부터는 그 스크립트만 실행한다. 대화로 시킬 때마다 토큰을 쓰지만, 스크립트는 공짜다.
내가 뭘 반복하는지조차 모르겠다면, 그것부터 클로드에게 물어보면 된다.
("이 세션에서 내가 반복적으로 시킨 작업이 있으면 스크립트로 만들어줘")

### 1-2. 안 쓰는 MCP는 꺼두기

MCP 서버의 도구 정의는 세션 시작 시 컨텍스트에 그대로 로드된다.
연결만 해두고 안 쓰는 MCP가 많으면 시작부터 컨텍스트를 낭비하는 셈이다.

- `/context` — 현재 컨텍스트가 어디에 얼마나 쓰이는지 확인
- `/mcp` — 서버 상태 확인, 필요할 때만 활성화

### 1-3. 세션은 작업 단위로 짧게

컨텍스트가 쌓일수록 신호 대비 잡음이 늘어나 응답 품질이 떨어진다.

- 작업이 끝나면 `/clear`로 초기화하고 새로 시작
- 이어가야 하는 긴 작업은 `/compact`로 요약해서 압축
- 하나의 세션에서 모든 걸 끝내려 하지 않기

### 1-4. CLAUDE.md는 적절한 길이 유지

CLAUDE.md는 매 세션 자동으로 로드되므로, 길어지면 그 자체가 상시 비용이 된다.
넣을 것만 넣는다:

- 자주 쓰는 빌드/테스트/린트 명령어
- 코드 스타일, 네이밍 규칙
- 프로젝트 구조에서 헷갈리기 쉬운 부분, 주의사항

대화 중 `#`으로 시작하는 메시지를 보내면 그 내용을 메모리에 추가할 수 있다.
주기적으로 열어보고 낡은 내용은 지운다. **CLAUDE.md도 리팩토링 대상이다.**

## 2. 지시하는 방법

### 2-1. 검증 기준을 미리 제시

"만들어줘"로 끝내지 말고, 무엇이 통과 기준인지 입출력으로 함께 준다.

> validateEmail 함수를 작성해줘
> - user@example.com → true
> - Invalid → false
> - user@.com → false

한 발 더 나가면 TDD가 된다: 테스트를 먼저 작성시키고, 실패하는 것을 확인한 뒤,
테스트를 통과하는 구현을 요청한다. 클로드가 스스로 결과를 검증할 수 있게 된다.

### 2-2. 탐색 → 계획 → 구현을 분리 (플랜 모드)

기획과 구현을 반드시 분리한다. `Shift+Tab` 두 번으로 플랜 모드에 진입하면
클로드는 파일을 수정하지 않고 읽기·조사만 하면서 계획을 세우고, 승인 후에 실행한다.

- 나쁜 예: "기존 로그인 기능에 OAuth 로그인을 구현해줘"
- 좋은 예:
  1. **탐색** — auth 코드를 읽고 기존 세션/로그인 처리 방식을 파악해줘
  2. **계획** — OAuth를 추가하려면 어떤 파일을 수정해야 하고 세션 흐름은 어떻게 될지 정리해줘
  3. **구현** — 계획대로 구현하고, 콜백 핸들러 테스트를 작성해서 통과를 확인해줘
  4. **커밋**

### 2-3. 클로드가 나를 인터뷰하게 만들기

기능 요구사항이나 UI/UX가 스스로도 확실치 않을 때는, 내가 설명하려 애쓰지 말고
클로드에게 질문을 시킨다. ("이 기능을 만들기 전에 확실히 해야 할 것들을 나에게 질문해줘")
인터뷰가 끝나면 그 내용을 스펙 문서로 정리시키고, 그 문서를 기준으로 구현을 시작한다.
**모호함이 코드가 되기 전에 문서 단계에서 제거된다.**

### 2-4. 잘못된 방향이면 빨리 개입

방향 수정은 빠를수록 싸다.

- `Esc` — 진행 중인 작업 즉시 중단
- `Esc` 두 번 — 이전 프롬프트로 돌아가 수정 후 다시 시도

틀린 코드가 쌓인 뒤에 고치는 것보다, 계획 단계에서 바로잡는 것이 훨씬 저렴하다.

### 2-5. 말보다 자료

UI 작업은 스크린샷이나 목업 이미지를 붙여넣고, 라이브러리 사용법은 문서 URL을 준다.
백 마디 설명보다 이미지 한 장이 정확하다.

## 3. 서브에이전트 활용

### 3-1. 격리된 컨텍스트가 최대 장점

서브에이전트는 자기만의 컨텍스트 윈도에서 작업하고 결과 요약만 메인 세션에 돌려준다.
파일 수십 개를 뒤지는 탐색 작업을 위임하면, 그 과정의 잡음이 메인 컨텍스트를
오염시키지 않는다. **서브에이전트는 클로드를 더 똑똑하게 만드는 게 아니라,
이미 있는 컨텍스트의 품질을 지켜준다.**

### 3-2. 내장 서브에이전트

Claude Code에는 기본 내장 에이전트가 있고, 상황에 따라 자동으로 위임된다.

| 에이전트 | 역할 |
|---|---|
| **Explore** | 읽기 전용, 경량 모델 기반의 빠른 코드베이스 탐색 |
| **Plan** | 플랜 모드에서 계획 수립 전 리서치 담당 |
| **general-purpose** | 탐색과 수정이 모두 필요한 복합 작업 담당 |

### 3-3. 커스텀 서브에이전트

`.claude/agents/`(프로젝트) 또는 `~/.claude/agents/`(전역)에
YAML 프론트매터가 붙은 마크다운 파일로 직접 정의할 수 있다.
역할별로 나눠두면 유용한 예시:

1. **code-reviewer** — 코드 품질·일관성 리뷰
2. **test-runner** — 테스트 실행 및 실패 분석
3. **debugger** — 에러 원인 추적
4. **security-auditor** — 보안 취약점 점검
5. **docs-writer** — 문서화 담당

서브에이전트 정의 자체도 클로드에게 작성시키는 게 빠르다.

### 3-4. Writer–Reviewer 멀티 세션 패턴

코드를 작성한 세션에게 리뷰까지 시키면, 자기가 쓴 코드에 대한 편향 때문에
스스로 만든 문제를 놓치는 경향이 있다. 새 세션(또는 리뷰어 서브에이전트)은
그 편향이 없어서 더 객관적으로 본다. 작성과 리뷰의 컨텍스트를 분리하자.

### 3-5. git worktree로 병렬 작업

독립적인 작업 여러 개를 동시에 진행할 때는 git worktree로 체크아웃을 분리하고
각 worktree에서 별도의 클로드 세션을 띄운다. 서로 충돌 없이 병렬로 진행된다.

## 4. 피해야 할 패턴

| 안티패턴 | 왜 나쁜가 |
|---|---|
| 한 세션에서 모든 걸 끝내려는 것 | 컨텍스트가 길어질수록 품질이 떨어진다 |
| CLAUDE.md 비대화 | 매 세션 로드되는 파일이 길면 그 자체가 낭비다 |
| 검증 기준 없는 지시 | "구현해줘" 한 줄은 클로드가 스스로 맞았는지 확인할 방법이 없다 |
| 확인 없는 자동 수락 남용 | 특히 계획 없이 큰 작업을 자동 수락으로 돌리는 것. 플랜 모드로 계획을 먼저 승인한 뒤 실행을 맡기는 편이 안전하다 |
