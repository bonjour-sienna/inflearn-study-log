## 0. 한눈에 보기

| 구분 | 정체 | 핵심 역할 | 실행 주체 | 적용 범위 |
|---|---|---|---|---|
| **MCP** | 개방형 표준 프로토콜 | 외부 시스템(DB, API, SaaS)과의 **연결** | 외부 서버 프로세스 | Claude Code, Claude Desktop, Claude.ai, 타사 AI 앱 전부 |
| **Skills** | SKILL.md가 든 폴더 | 지식·절차를 Claude에게 **가르침** (필요할 때만 로드) | Claude (모델이 판단해 사용) | Claude Code, Claude.ai, Claude API (Agent Skills 오픈 표준) |
| **Hooks** | 라이프사이클 이벤트 핸들러 | 특정 시점에 **반드시** 실행/차단되는 **강제** 규칙 | 결정론적 (모델 판단과 무관) | Claude Code, Agent SDK |
| **Plugins** | 배포 패키지 | 위 모든 것 + 에이전트를 묶어 **공유/배포** | — (컨테이너 역할) | Claude Code |

관계를 한 문장으로 정리하면: **MCP는 손(외부 도구), Skill은 머리(지식과 절차), Hook은 규율(강제 규칙), Plugin은 상자(배포 단위)** 입니다. 플러그인 하나가 스킬·훅·MCP 서버·서브에이전트를 전부 담을 수 있고, 스킬의 frontmatter에 훅을 넣을 수도 있으며, 훅이 MCP 도구 호출(`mcp__서버__도구`)을 감시하거나 차단할 수도 있습니다.

## 1. MCP (Model Context Protocol)

### 1-1. 이론

MCP는 LLM 애플리케이션이 외부 데이터와 기능에 안전하고 표준화된 방식으로 접근하게 해주는 **개방형 프로토콜**입니다. "웹 API와 비슷하지만 사람이 아니라 LLM을 위해 설계된 것"으로 이해하면 됩니다. Anthropic이 만들었지만 열린 표준이라 Claude 외의 도구들도 채택하고 있습니다.

**아키텍처 (3계층)**

```
Host (Claude Code, Claude Desktop 등 AI 앱)
 └── Client (호스트 내부, 서버와 1:1 연결 유지)
      └── Server (기능을 제공하는 별도 프로그램 — 여러분이 만드는 것)
```

**서버가 제공하는 3가지 기본 요소**

| 요소 | 설명 | 예시 |
|---|---|---|
| **Tools** | 모델이 호출하는 함수 (동작 수행) | `create_issue()`, `query_db()` |
| **Resources** | 모델/앱이 읽는 데이터 (URI로 식별) | `file://...`, `config://app` |
| **Prompts** | 재사용 가능한 프롬프트 템플릿 | 코드 리뷰 템플릿 |

**전송 방식 (Transport)**

| 방식 | 동작 | 용도 |
|---|---|---|
| **stdio** (기본) | 로컬 서브프로세스로 실행, stdin/stdout으로 JSON-RPC 통신 | 로컬 도구, 개인 스크립트 |
| **HTTP** (Streamable HTTP) | 원격 URL에 접속 | 호스팅 서버 (GitHub, Sentry 등), OAuth 지원 |
| **SSE** | 구형 원격 방식 | 사용 중단 방향 — 신규는 HTTP 사용 |

**중요한 함정**: stdio 서버에서 `print()`/`console.log()`로 stdout에 로그를 찍으면 JSON-RPC 메시지가 깨져 서버가 망가집니다. 로그는 반드시 **stderr**로 보내세요 (Python `logging`, JS `console.error()`).

### 1-2. 만드는 법 예시 (Python + FastMCP)

**설치**

```bash
pip install "mcp[cli]"      # 공식 SDK (FastMCP 포함)
# 또는 uv add "mcp[cli]"
```

**server.py — 도구·리소스·프롬프트를 모두 가진 최소 서버**

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-tools")   # 서버 이름

# ① Tool: 함수 시그니처(타입 힌트)와 docstring에서 스키마가 자동 생성됨
@mcp.tool()
def add(a: int, b: int) -> int:
    """두 수를 더합니다."""
    return a + b

@mcp.tool()
async def fetch_weather(city: str) -> str:
    """도시의 날씨를 조회합니다."""
    # httpx 등으로 실제 API 호출...
    return f"{city}: 맑음, 27도"

# ② Resource: URI로 읽을 수 있는 데이터
@mcp.resource("config://app")
def get_config() -> str:
    """앱 설정을 반환합니다."""
    return "environment: production"

# ③ Prompt: 재사용 프롬프트 템플릿
@mcp.prompt()
def review_code(code: str) -> str:
    return f"다음 코드를 리뷰해 주세요:\n\n{code}"

if __name__ == "__main__":
    mcp.run(transport="stdio")   # HTTP로 띄우려면 transport="streamable-http"
```

**Claude Code에 등록**

```bash
# 로컬 stdio 서버 (기본). `--` 뒤가 서버 실행 명령
claude mcp add my-tools -- python /path/to/server.py

# 원격 HTTP 서버
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# 팀 공유용: 프로젝트 루트에 .mcp.json 생성 → git 커밋 가능
claude mcp add --scope project my-tools -- python server.py

# 환경 변수 전달
claude mcp add my-db -e API_KEY=xxx -- npx -y @modelcontextprotocol/server-postgres postgresql://localhost/db

claude mcp list      # 등록된 서버와 연결 상태 확인
claude mcp remove my-tools
```

세션 안에서는 `/mcp`로 서버 상태 확인과 OAuth 인증을 처리합니다.

**스코프 3종**

| 스코프 | 저장 위치 | 공유 |
|---|---|---|
| `local` (기본) | `~/.claude.json` (프로젝트별 항목) | 본인만, 해당 프로젝트만 |
| `project` | 프로젝트 루트 `.mcp.json` | git 커밋으로 팀 공유 |
| `user` | `~/.claude.json` | 본인의 모든 프로젝트 |

**.mcp.json 직접 작성 예시**

```json
{
  "mcpServers": {
    "github": { "type": "http", "url": "https://api.githubcopilot.com/mcp/" },
    "my-tools": { "command": "python", "args": ["server.py"] }
  }
}
```

## 2. Skills (Agent Skills)

### 2-1. 이론

스킬은 **SKILL.md 파일이 든 폴더**입니다. 지침(마크다운) + 선택적으로 스크립트·템플릿·참고 문서를 함께 담습니다. 같은 지침을 채팅에 반복해서 붙여넣고 있거나, CLAUDE.md의 한 섹션이 "사실"이 아니라 "절차"로 커졌다면 스킬로 뺄 때입니다. CLAUDE.md와 달리 스킬 본문은 **사용될 때만 로드**되므로 긴 참고 자료를 넣어도 평소엔 토큰 비용이 거의 없습니다.

Claude Code의 스킬은 [Agent Skills 오픈 표준](https://agentskills.io)을 따르며, Claude.ai와 Claude API(Skills API)에서도 같은 형식이 통합니다. 또한 **커스텀 슬래시 커맨드가 스킬로 통합**되었습니다. `.claude/commands/deploy.md`와 `.claude/skills/deploy/SKILL.md`는 둘 다 `/deploy`를 만들며 동일하게 동작합니다 (스킬 쪽이 지원 파일, 자동 로드 등 기능이 더 많아 권장됨).

**핵심 개념: 점진적 공개 (Progressive Disclosure) — 3단계 로딩**

| 단계 | 로드 시점 | 내용 |
|---|---|---|
| 1. 메타데이터 | 항상 컨텍스트에 존재 | frontmatter의 `name` + `description` |
| 2. SKILL.md 본문 | 스킬이 트리거될 때 | 지침 전체 (500줄 이하 권장) |
| 3. 번들 리소스 | 필요할 때만 | references/, scripts/ 등 — 스크립트는 로드 없이 실행만 가능 |

Claude는 1단계의 description만 보고 "이 스킬을 쓸지"를 판단합니다. 그래서 **description이 트리거 품질을 좌우**합니다.

**저장 위치와 우선순위**

| 위치 | 경로 | 적용 범위 |
|---|---|---|
| 엔터프라이즈 | 관리 설정 (managed settings) | 조직 전체 |
| 개인 | `~/.claude/skills/<이름>/SKILL.md` | 내 모든 프로젝트 |
| 프로젝트 | `.claude/skills/<이름>/SKILL.md` | 해당 프로젝트 (git 커밋 가능) |
| 플러그인 | `<plugin>/skills/<이름>/SKILL.md` | 플러그인 활성 시 (`플러그인명:스킬명`으로 네임스페이스 분리) |

이름이 겹치면 엔터프라이즈 > 개인 > 프로젝트 순으로 우선합니다.

**주요 frontmatter 필드** (모두 선택 사항, `description`만 강력 권장)

| 필드 | 역할 |
|---|---|
| `name` | 표시 이름 (기본값: 디렉터리명). 디렉터리명이 곧 `/명령어`가 됨 |
| `description` | 무엇을 하고 **언제 쓰는지**. Claude의 자동 트리거 판단 근거 |
| `disable-model-invocation: true` | Claude의 자동 실행 금지 — 사용자가 `/이름`으로만 실행 (배포, 커밋 등 부작용 있는 작업용) |
| `user-invocable: false` | `/` 메뉴에서 숨김 — Claude만 사용 (배경 지식용) |
| `allowed-tools` | 스킬 실행 턴 동안 승인 없이 쓸 수 있는 도구 사전 허용 |
| `context: fork` + `agent` | 서브에이전트(격리 컨텍스트)에서 실행. `agent: Explore` 등 지정 |
| `model` / `effort` | 스킬 활성 중 사용할 모델/노력 수준 오버라이드 |
| `paths` | 특정 파일 패턴 작업 시에만 활성화 |

**본문에서 쓸 수 있는 치환**: `$ARGUMENTS`(전체 인자), `$0` `$1`(위치 인자), `${CLAUDE_SKILL_DIR}`(스킬 폴더 경로), `${CLAUDE_PROJECT_DIR}`. 그리고 `` !`명령` `` 구문은 **동적 컨텍스트 주입** — Claude가 보기 전에 셸 명령을 먼저 실행해 그 출력으로 치환합니다.

### 2-2. 만드는 법 예시

**예시 A — 기본형: 미커밋 변경 요약 스킬** (공식 문서의 첫 스킬 예제)

```bash
mkdir -p ~/.claude/skills/summarize-changes
```

`~/.claude/skills/summarize-changes/SKILL.md`:

```yaml
---
description: 커밋하지 않은 변경 사항을 요약하고 위험 요소를 표시합니다. 사용자가 무엇이 바뀌었는지 묻거나, 커밋 메시지를 원하거나, diff 리뷰를 요청할 때 사용.
---
## 현재 변경 사항
!`git diff HEAD`

## 지침
위 변경 사항을 2~3개 항목으로 요약한 뒤, 누락된 에러 처리·하드코딩 값·업데이트가
필요한 테스트 같은 위험 요소를 나열하세요. diff가 비어 있으면 변경 사항이 없다고 답하세요.
```

테스트: git 프로젝트에서 파일을 하나 수정한 뒤 `claude` 실행 → "뭘 바꿨지?"라고 묻거나(자동 트리거) `/summarize-changes`로 직접 실행.

**예시 B — 실행형: 배포 스킬 (사용자만 실행 가능 + 인자 + 도구 사전 허용)**

`.claude/skills/deploy/SKILL.md`:

```yaml
---
name: deploy
description: 애플리케이션을 프로덕션에 배포
disable-model-invocation: true
allowed-tools: Bash(npm run *) Bash(git *)
---
$ARGUMENTS 환경에 배포:
1. 테스트 스위트 실행
2. 애플리케이션 빌드
3. 배포 대상에 푸시
4. 배포 성공 검증
```

`/deploy staging`처럼 실행하면 `$ARGUMENTS`가 `staging`으로 치환됩니다. `disable-model-invocation: true` 덕분에 "코드가 준비된 것 같으니 배포하자"라고 Claude가 멋대로 실행하는 일이 없습니다.

**예시 C — 지원 파일이 있는 구조**

```text
my-skill/
├── SKILL.md          # 필수 — 개요와 네비게이션
├── reference.md      # 상세 API 문서 (필요할 때만 로드)
├── examples.md       # 사용 예시
└── scripts/
    └── helper.py     # 실행만 되고 컨텍스트에 로드되지 않음
```

SKILL.md 안에서 "상세 API는 [reference.md](reference.md) 참고"처럼 **언제 어떤 파일을 읽을지 명시**해야 Claude가 활용합니다. 스크립트는 `` !`python ${CLAUDE_SKILL_DIR}/scripts/helper.py` `` 또는 본문 지침으로 실행시킵니다.

**작성 팁**

- description에는 사용자가 실제로 말할 법한 **키워드**를 넣으세요 (트리거 정확도의 핵심).
- SKILL.md는 500줄 이하로. 길어지면 references/로 분리.
- 명령형 문체로, "왜 중요한지"를 설명하는 방식이 무거운 MUST 나열보다 잘 동작합니다.
- 평가와 개선은 공식 `skill-creator` 플러그인이 자동화해 줍니다: `/plugin install skill-creator@claude-plugins-official` 후 "내 스킬을 평가해 줘"라고 요청 — 테스트 케이스 작성, 스킬 유/무 비교 실행, 채점, description 튜닝까지 루프를 돌려 줍니다.

## 3. Hooks

### 3-1. 이론

훅은 Claude Code 라이프사이클의 특정 시점에 실행되는 **사용자 정의 명령**입니다. 핵심 가치는 **결정론**: "LLM이 해 주기를 기대"하는 게 아니라 "반드시 실행/차단"됩니다. CLAUDE.md는 설득하고, 권한(permissions)은 정적으로 거르고, 훅은 동적으로 강제하고 반응합니다 — 세 층이 함께 쓰입니다.

**주요 이벤트** (현재 30여 개 — 자주 쓰는 것 위주)

| 이벤트 | 발생 시점 | 대표 용도 |
|---|---|---|
| `SessionStart` | 세션 시작/재개/압축 후 | 컨텍스트 주입, 환경 변수 로드 |
| `UserPromptSubmit` | 프롬프트 제출 직후, Claude 처리 전 | 컨텍스트 추가, 프롬프트 검열 |
| `PreToolUse` | 도구 호출 **직전** | **차단 가능** — 보호 파일, 위험 명령 |
| `PermissionRequest` | 권한 다이얼로그 표시 시점 | 자동 승인/거부 |
| `PostToolUse` | 도구 호출 성공 후 | 자동 포맷, 로깅 (실행 취소는 불가) |
| `Notification` | 알림 발생 (입력 대기 등) | 데스크톱/슬랙 알림 |
| `Stop` | Claude가 응답을 마칠 때 | 완료 검증, exit 2로 "계속 일해" 강제 |
| `SubagentStart` / `SubagentStop` | 서브에이전트 시작/종료 | 서브에이전트 감시 |
| `PreCompact` / `PostCompact` | 컨텍스트 압축 전/후 | 압축 후 핵심 컨텍스트 재주입 |
| `SessionEnd` | 세션 종료 | 임시 파일 정리 |

그 외 `ConfigChange`(설정 파일 변경 감사/차단), `FileChanged`(파일 감시), `CwdChanged`(디렉터리 이동 시 direnv 재로드), `PostToolUseFailure`, `Setup` 등이 있습니다.

**통신 프로토콜** — 훅 스크립트는 stdin/stdout/stderr/종료 코드로 대화합니다.

1. 이벤트 발생 → 이벤트 데이터가 **JSON으로 stdin**에 들어옴 (`session_id`, `cwd`, `tool_name`, `tool_input` 등)
2. 스크립트가 판단 후 **종료 코드**로 응답:

| 종료 코드 | 의미 |
|---|---|
| `0` | 이견 없음, 정상 진행. (`UserPromptSubmit`, `SessionStart` 등에서는 stdout이 Claude 컨텍스트에 추가됨) |
| `2` | **차단**. stderr에 쓴 내용이 Claude에게 피드백으로 전달되어 방향을 수정하게 함 |
| 기타 | 진행하되 에러 표시 |

3. 더 세밀한 제어가 필요하면 **exit 0 + stdout에 구조화 JSON**:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "grep 대신 rg를 사용하세요"
  }
}
```

`permissionDecision`은 `allow`(프롬프트 생략) / `deny`(차단+사유 전달) / `ask`(사용자에게 묻기). 단, 훅의 `allow`가 설정의 deny 규칙을 이기지는 못합니다 — **훅은 조일 수는 있어도 풀 수는 없습니다**. 반대로 `PreToolUse`의 deny는 `bypassPermissions` 모드에서도 통합니다.

**matcher** — 이벤트별 필터. 도구 이벤트는 도구 이름(`Bash`, `Edit|Write`, `mcp__github__.*`), `SessionStart`는 시작 방식(`startup`, `compact`) 등. 비워 두면 모든 발생에 실행. `if` 필드로 권한 규칙 문법(`Bash(git *)`)을 써서 인자까지 필터링할 수도 있습니다.

**훅 타입 5종**

| 타입 | 동작 |
|---|---|
| `command` (기본) | 셸 명령 실행 |
| `http` | 이벤트 JSON을 URL로 POST |
| `mcp_tool` | 연결된 MCP 서버의 도구 호출 |
| `prompt` | 단발 LLM 호출(기본 Haiku)로 `{"ok": true/false, "reason": ...}` 판단 |
| `agent` | 파일 읽기·명령 실행이 가능한 서브에이전트로 다중 턴 검증 (실험적) |

**설정 위치**: `~/.claude/settings.json`(전역), `.claude/settings.json`(프로젝트·공유), `.claude/settings.local.json`(개인·비공유), 관리 정책, 플러그인의 `hooks/hooks.json`, 스킬/에이전트 frontmatter의 `hooks` 필드. 확인은 `/hooks` 메뉴, 디버깅은 `claude --debug`.

### 3-2. 만드는 법 예시

**예시 A — 편집 후 자동 포맷 (PostToolUse)**

`.claude/settings.json`:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}
```

Claude가 `Edit`/`Write`로 파일을 고칠 때마다 stdin의 JSON에서 파일 경로를 뽑아 Prettier를 실행합니다. "포맷 좀 돌려 줘"를 다시는 말할 필요가 없습니다.

**예시 B — 보호 파일 편집 차단 (PreToolUse + 스크립트 + exit 2)**

`.claude/hooks/protect-files.sh`:

```bash
#!/bin/bash
INPUT=$(cat)                                    # stdin으로 이벤트 JSON 수신
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

PROTECTED=(".env" "package-lock.json" ".git/")
for pattern in "${PROTECTED[@]}"; do
  if [[ "$FILE_PATH" == *"$pattern"* ]]; then
    echo "차단됨: $FILE_PATH 는 보호 패턴 '$pattern'에 해당" >&2   # stderr → Claude 피드백
    exit 2                                                        # 2 = 차단
  fi
done
exit 0
```

```bash
chmod +x .claude/hooks/protect-files.sh
```

`.claude/settings.json`에 등록:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/protect-files.sh"
          }
        ]
      }
    ]
  }
}
```

**예시 C — 작업 완료 검증 (Stop + prompt 타입 훅)**

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "요청된 모든 작업이 완료되었는지 확인하세요. 미완료라면 {\"ok\": false, \"reason\": \"남은 작업 내용\"} 으로 응답하세요."
          }
        ]
      }
    ]
  }
}
```

Claude가 멈추려 할 때 모델이 완료 여부를 판단하고, `ok: false`면 `reason`을 다음 지시로 삼아 계속 일합니다. (무한 루프 방지를 위해 8회 연속 차단 시 강제 종료됩니다.)

## 4. Plugins

### 4-1. 이론

플러그인은 **여러 확장 컴포넌트를 하나의 디렉터리로 묶어 배포하는 단위**입니다. 담을 수 있는 것: 스킬, 서브에이전트, 훅, MCP 서버, LSP 서버 등. "혼자 쓸 땐 `.claude/` 디렉터리, 팀·커뮤니티에 나눠 줄 땐 플러그인" — 기능 차이가 아니라 **배포 방식의 선택**입니다.

**표준 디렉터리 구조** — 매우 중요한 규칙: `.claude-plugin/` 안에는 **plugin.json만** 들어가고, 나머지 컴포넌트는 전부 **플러그인 루트**에 둡니다.

```text
my-plugin/
├── .claude-plugin/
│   └── plugin.json        # 매니페스트 (여기엔 이 파일만!)
├── skills/                # 스킬 → /my-plugin:스킬명 으로 네임스페이스됨
│   └── review/
│       └── SKILL.md
├── commands/              # (구) 슬래시 커맨드 .md — 스킬로 통합됐지만 여전히 동작
├── agents/                # 서브에이전트 정의 .md
│   └── security-expert.md
├── hooks/
│   └── hooks.json         # settings.json의 hooks 블록과 동일 형식
├── .mcp.json              # 번들 MCP 서버
└── README.md
```

특징 몇 가지: 매니페스트는 사실 선택 사항이며 없으면 디렉터리명으로 자동 인식됩니다. 플러그인 컴포넌트는 `플러그인명:이름`으로 네임스페이스되어 다른 레벨과 충돌하지 않습니다. 설치 시 플러그인 디렉터리가 캐시로 **복사**되므로 `../` 같은 바깥 경로 참조는 깨집니다 — 플러그인 내부 파일은 `${CLAUDE_PLUGIN_ROOT}` 변수로 참조하세요. 반대로 스킬 폴더에 `.claude-plugin/plugin.json`을 추가하면 그 스킬 폴더 자체가 플러그인으로 승격되어 훅·MCP까지 담을 수 있습니다.

**마켓플레이스** — 플러그인 배포 카탈로그. git 저장소 루트에 `.claude-plugin/marketplace.json`을 두면 됩니다. 중앙 검색, 버전 추적, 자동 업데이트를 제공하며 소스로 git 저장소·GitHub·로컬 경로 등을 쓸 수 있습니다. Anthropic 공식 마켓플레이스(`claude-plugins-official`)는 모든 설치에 기본 포함됩니다.

### 4-2. 만드는 법 예시

**1단계 — 매니페스트 작성**

`my-plugin/.claude-plugin/plugin.json`:

```json
{
  "name": "my-plugin",
  "version": "0.1.0",
  "description": "코드 리뷰 자동화 묶음: 리뷰 스킬 + 포맷 훅 + 사내 API MCP",
  "author": { "name": "홍길동", "email": "hong@example.com" },
  "keywords": ["review", "automation"]
}
```

**2단계 — 컴포넌트 채우기**

스킬: `my-plugin/skills/review/SKILL.md`

```yaml
---
description: 우리 팀 컨벤션에 따라 코드를 리뷰합니다. PR 리뷰, 코드 검토 요청 시 사용.
---
다음 순서로 리뷰하세요:
1. 보안 취약점 (하드코딩된 비밀 값, 인젝션)
2. 에러 처리 누락
3. 팀 네이밍 컨벤션 (checklist.md 참고)
```

훅: `my-plugin/hooks/hooks.json` — 플러그인 내부 스크립트는 `${CLAUDE_PLUGIN_ROOT}`로 참조

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "${CLAUDE_PLUGIN_ROOT}/scripts/format.sh" }
        ]
      }
    ]
  }
}
```

MCP 서버: `my-plugin/.mcp.json`

```json
{
  "mcpServers": {
    "team-api": { "type": "http", "url": "https://mcp.example.com/mcp" }
  }
}
```

서브에이전트: `my-plugin/agents/security-expert.md`

```yaml
---
name: security-expert
description: 보안 관점 코드 검토 전문 서브에이전트
---
당신은 보안 검토 전문가입니다. OWASP Top 10을 기준으로...
```

**3단계 — 마켓플레이스로 배포**

배포용 저장소 루트에 `.claude-plugin/marketplace.json`:

```json
{
  "name": "my-team-marketplace",
  "owner": { "name": "홍길동" },
  "plugins": [
    {
      "name": "my-plugin",
      "source": "./plugins/my-plugin",
      "version": "0.1.0",
      "description": "코드 리뷰 자동화 묶음",
      "category": "development"
    }
  ]
}
```

`source`는 같은 저장소 내 상대 경로 외에 `{"source": "github", "repo": "org/other-repo"}`처럼 다른 저장소를 가리킬 수도 있어, 카탈로그와 플러그인을 독립적으로 관리할 수 있습니다. 저장소를 GitHub에 공개로 푸시하면 끝입니다.

**4단계 — 설치 (사용자 쪽)**

```text
/plugin marketplace add your-org/your-repo      # 마켓플레이스 등록 (로컬 경로도 가능)
/plugin install my-plugin@my-team-marketplace   # 설치
/plugin                                          # 대화형 브라우저 UI
/reload-plugins                                  # 스킬 외 컴포넌트 변경 반영
```

공식 마켓플레이스에서 설치할 때의 형태: `/plugin install skill-creator@claude-plugins-official`. 개발 중 로컬 테스트는 마켓플레이스에 로컬 경로를 추가해 설치하는 방식이 편합니다. 공식 디렉터리 등재를 원하면 플러그인 제출 폼(claude.ai/settings/plugins/submit)으로 심사를 신청합니다.

## 5. 무엇을 언제 쓸까 — 선택 가이드

| 하고 싶은 것 | 선택 | 이유 |
|---|---|---|
| 반복해서 알려 주는 절차·컨벤션·도메인 지식 | **Skill** | 필요할 때만 로드, Claude가 스스로 판단해 적용 |
| "반드시" 실행되거나 "절대" 안 되는 규칙 (포맷 강제, 위험 명령 차단, 감사 로그) | **Hook** | 결정론적 — 모델이 우회할 수 없음 |
| 외부 시스템 연결 (DB, GitHub, Slack, 사내 API) | **MCP** | 표준 프로토콜, 앱 간 재사용 |
| 위의 조합을 팀/커뮤니티에 한 번에 배포 | **Plugin** | 버전 관리 + 마켓플레이스 배포 |
| 항상 알아야 하는 짧은 사실 (빌드 명령, 프로젝트 개요) | CLAUDE.md | 항상 컨텍스트에 있음 (길어지면 스킬로 분리) |
| 격리된 컨텍스트에서 돌릴 작업 | 서브에이전트 (또는 스킬 `context: fork`) | 메인 대화 오염 방지 |

기억해 둘 경계선 하나: **스킬은 "설득"이고 훅은 "강제"입니다.** 스킬에 "항상 테스트를 돌려라"라고 써도 모델이 건너뛸 수 있지만, `Stop` 훅으로 테스트 통과를 검증하면 건너뛸 수 없습니다. 어겨서는 안 되는 규칙은 훅(또는 권한 설정)에, 판단이 필요한 지식은 스킬에 두세요.
