## 오늘 배운 것 — Claude Code에서 실제로 병렬 실행하는 방법

먼저 구분할 것: **자연어 지시 · `@mention` · `--agent` 플래그는 에이전트를 "호출"하는
방법일 뿐, 그 자체가 병렬 실행은 아니다.** 실제 병렬화 수단은 아래 세 가지다.

### ① 한 세션에서 여러 서브에이전트 동시 실행

```
별도의 서브에이전트를 사용해서 다음을 병렬로 조사해줘.
1. 인증 모듈의 토큰 갱신 구조
2. 데이터베이스 트랜잭션 처리
3. 외부 API 예외 처리
모두 완료되면 결과를 종합해줘.
```

```
Main Claude
 ├─ Subagent A: 인증 조사
 ├─ Subagent B: DB 조사
 └─ Subagent C: API 조사
```

각 서브에이전트가 독립된 컨텍스트에서 탐색하고 Claude가 결과를 종합한다. 조사 경로가
서로 의존하지 않을 때 가장 효과적이며, 공식 문서도 인증·DB·API 모듈 병렬 조사를 대표
예시로 든다.

실행 방식에 대한 정확한 표현:

- Claude가 작업 성격에 따라 **포그라운드(블로킹)** 또는 **백그라운드(동시 실행)** 를
  결정한다. "백그라운드에서 실행해줘"라고 요청하거나, 실행 중인 작업에 `Ctrl+B`를 눌러
  백그라운드로 보낼 수 있다. **항상 백그라운드가 기본값인 것은 아니다.**
- 특정 서브에이전트를 항상 백그라운드로 돌리려면 frontmatter에 `background: true`.
- 현재 세션의 백그라운드 작업은 `/tasks`로 확인·중지, 실행 중인 서브에이전트는
  `/agents`의 **Running 탭**에서 확인.
- 서브에이전트들이 각자 상세한 결과를 반환하면 **메인 컨텍스트가 상당히 소모**된다.

### ② 중첩 서브에이전트는 **지원되지 않음**

서브에이전트는 다른 서브에이전트를 생성할 수 없다. 무한 중첩 방지를 위한 제약이고,
Plan 서브에이전트 설명에도 명시돼 있다. 중첩 위임이 필요하면 **Skills를 쓰거나 메인
대화에서 서브에이전트를 체이닝**하라고 문서가 안내한다.

따라서 "리뷰 총괄 → 보안/성능/테스트 하위 에이전트" 같은 계층 구조가 필요하면:

- 메인이 직접 보안·성능·테스트 서브에이전트를 병렬로 띄우고 종합하거나 (**평평한 구조**),
- 순차 의존이 있으면 "code-reviewer로 문제를 찾고, optimizer로 고쳐줘"처럼 **체이닝**하거나,
- 진짜 계층·협업이 필요하면 **Agent Teams**를 쓴다.

### ③ Agent Teams (실험적 기능)

```
Lead (메인 세션)
 ├─ Backend teammate
 ├─ Frontend teammate
 └─ Test teammate
```

기본 비활성화이며 settings.json이나 환경변수에 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`를
추가해야 켜진다. 리드 세션이 작업을 조율·배분·종합하고, 각 팀메이트는 자기만의 컨텍스트
창에서 독립적으로 일하면서 **서로 직접 통신**한다.

- **서브에이전트와의 결정적 차이가 이 통신이다** — 서브에이전트는 결과를 메인에 보고만
  할 뿐 서로 대화하지 못한다.
- **리드는 고정**이다. 메인 세션이 수명 내내 리드이며 팀메이트를 리드로 승격하거나
  리더십을 넘길 수 없다.
- 지속적인 병렬 작업이 필요하거나 컨텍스트 창을 초과하는 작업이면 각 워커가 독립
  컨텍스트를 갖는 agent teams가 적합하다.
-  **파일 충돌 주의**: agent teams는 팀메이트를 worktree로 격리하지 **않으므로**,
  각 팀메이트가 서로 다른 파일을 담당하도록 작업을 분할해야 한다.

### ④ Worktree 다중 세션

```bash
claude --worktree feature-auth
claude --worktree fix-payment
claude --worktree add-tests
```

`--worktree`(`-w`)를 주면 저장소 루트의 `.claude/worktrees/<이름>/` 아래에 격리된
worktree가 `worktree-<이름>` 브랜치로 생성되고 그 안에서 Claude가 시작된다. 다른
터미널에서 다른 이름으로 다시 실행하면 두 번째 격리 세션이 뜬다. 이건 서브에이전트
병렬이 아니라 **독립 세션 병렬**이며, 파일 수정 충돌을 **파일시스템 수준**에서 막아준다.

추가로 알아두면 좋은 것:

- 서브에이전트도 frontmatter에 `isolation: worktree`를 설정하면 각자 임시 worktree에서
  실행돼 병렬 편집이 충돌하지 않고, 변경 없이 끝나면 자동 정리된다. → **①번 방식에도
  worktree 격리를 결합할 수 있다.**
- 데스크톱 앱은 새 세션마다 자동으로 worktree를 생성한다.
- 백그라운드로 돌린 독립 세션들은 `claude agents`로 여는 **agent view**(리서치 프리뷰)에서
  한 화면으로 모아 볼 수 있다.

## 핵심 정리

| 방식 | 실제 병렬 | 컨텍스트 | 파일 충돌 격리 | 적합한 작업 |
|---|---|---|---|---|
| 백그라운드 서브에이전트 | 가능 | 메인 세션 아래 각각 분리 | 기본 없음, `isolation: worktree`로 가능 | 독립적인 조사·검증 |
| Agent Teams (실험적) | 가능 | 팀메이트별 독립 세션 | 없음 → 파일 분담 필요 | 장시간 협업·통신이 필요한 구현 |
| Worktree 다중 세션 | 가능 | 완전히 독립 | 브랜치·디렉터리 격리 | 동시에 여러 기능 수정 |

## 💡 결론

호출 방법은 3가지(자연어, `@mention`, `--agent`)이고, **실제 병렬화는 백그라운드
서브에이전트 · agent teams · worktree 독립 세션**으로 구성한다. **중첩 서브에이전트는
현재 지원되지 않으며**, 계층적 위임이 필요하면 체이닝이나 agent teams로 대체한다.
