## 오늘 배운 것 — 바이브 코딩 (Vibe Coding)

## 1. 바이브 코딩이란?

- **유래**: 안드레이 카파시(Andrej Karpathy)가 묘사한 방식 — 음성으로 지시하고, diff를 읽지 않고 무조건 수락하며, 코드를 이해하는 것을 포기한 채 개발하는 것
- **핵심 정의**: "AI가 다 해주는 것"이 아니라 **작성된 코드를 읽지 않는 것**이 바이브 코딩이다
- 바이브 코딩은 어디까지나 **출발점**이다 → 그 결과물을 그대로 프로덕트에 올리면 안 된다
- **효과적인 상황**: MVP를 빨리 만들어 고객에게 보여줘야 할 때, 프로토타이핑을 빨리 해야 할 때
- **결론**: 바이브 코딩을 적재적소에 쓸 줄 아는 사람이 되어야 한다

> **보충**: 카파시는 OpenAI 창립 멤버이자 전 테슬라 AI 총괄로, 2025년 2월 X(트위터) 게시글에서 이 용어를 처음 사용했어요. "코드가 존재한다는 것조차 잊고 분위기(vibe)에 완전히 몸을 맡긴다"는 표현에서 나온 이름입니다.


## 2. 품질 차이는 어디서 오는가?

같은 AI를 써도 결과물이 갈리는 지점 5가지:

1. 스펙·기술 스택·제약사항을 정리해서 전달한다
2. 에러가 발생하면 원인 분석을 같이 해서 적절한 방향을 지시한다
3. 컨텍스트 길이를 적절히 관리한다
4. 기술 스택을 미리 지정해서 일관성을 확보한다
5. 스키마 → API → UI → 테스트 단계를 분리해서 구현을 위임한다


## 3. 바이브 코딩, 어디에 쓰나?

1. **빠른 프로토타이핑** — 프로토타입의 목적은 검증이다. 이 개념이 기술적으로 구현 가능한지, 사용자가 인터페이스를 이해하는지 알아보는 단계이므로 속도가 중요
2. **MVP** — 시장 반응 테스트용 최소 제품. 검증할 질문: 핵심 기능이 동작하는가? 사용자가 돈을 낼 의향이 있는가? 이 비즈니스가 성립하는가? 첫 사용자 10명을 확보할 수 있는가?
3. **내부 도구** — 완성도보다 기능이 중요하다 (대시보드, 슬랙봇, 데이터 마이그레이션 도구 등)
4. **학습·탐색** — 새로운 기술을 배우거나 기능을 빠르게 탐색하고 싶을 때
