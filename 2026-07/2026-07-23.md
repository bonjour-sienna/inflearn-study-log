## 1. 바이브 코딩이란?

- **유래**: 안드레이 카파시(Andrej Karpathy)가 묘사한 방식 — 음성으로 지시하고, diff를 읽지 않고 무조건 수락하며, 코드를 이해하는 것을 포기한 채 개발하는 것
- **핵심 정의**: "AI가 다 해주는 것"이 아니라 **작성된 코드를 읽지 않는 것**이 바이브 코딩이다
- 바이브 코딩은 어디까지나 **출발점**이다 → 그 결과물을 그대로 프로덕트에 올리면 안 된다
- **효과적인 상황**: MVP를 빨리 만들어 고객에게 보여줘야 할 때, 프로토타이핑을 빨리 해야 할 때
- **결론**: 바이브 코딩을 적재적소에 쓸 줄 아는 사람이 되어야 한다

> **보충**: 카파시는 OpenAI 창립 멤버이자 전 테슬라 AI 총괄로, 2025년 2월 X(트위터) 게시글에서 이 용어를 처음 사용했어요. "코드가 존재한다는 것조차 잊고 분위기(vibe)에 완전히 몸을 맡긴다"는 표현에서 나온 이름입니다.

---

## 2. 품질 차이는 어디서 오는가?

같은 AI를 써도 결과물이 갈리는 지점 5가지:

1. 스펙·기술 스택·제약사항을 정리해서 전달한다
2. 에러가 발생하면 원인 분석을 같이 해서 적절한 방향을 지시한다
3. 컨텍스트 길이를 적절히 관리한다
4. 기술 스택을 미리 지정해서 일관성을 확보한다
5. 스키마 → API → UI → 테스트 단계를 분리해서 구현을 위임한다

> **보충 (컨텍스트 관리)**: 대화가 길어질수록 모델이 초반 지시를 잊거나 품질이 떨어져요. 작업 단위가 바뀌면 새 세션을 열고, 필요한 맥락은 문서로 넘겨주는 방식이 유리합니다. 5번의 단계 분리가 효과적인 이유도 결국 한 번에 다루는 컨텍스트를 작게 유지하기 때문이에요.

## 3. 바이브 코딩의 좋은 습관 3가지

1. **PRD 작성** — 무엇을 만들지 문서화한다. AI가 추측할 영역을 줄여 일관된 방향을 확보한다
   - 나의 적용 사례: 코드 문제점을 찾고 수정하는 단계도 이 문서에서 시작한다. ① 현재 코드의 문제를 상세히 분석해 달라고 요청 → ② 그 주제로 대화 → ③ 대화 내용을 요약·정리해 문서화. 이 과정에서 코드를 수정하는 이유와 개발 단계까지 도출된다
2. **작은 단위로 쪼개기** — 큰 기능은 작은 단위로 나누어 진행하고, 각 단계별로 검증한다
3. **검증과 피드백** — 결과를 검증하고 피드백을 준다

**2026년 현재의 변화:**

- 모델이 충분히 똑똑해져서 결과를 알아서 검증하고 에러를 잘 수정할 수 있다
- 사람의 역할은 LLM이 개발을 잘할 수 있게 환경을 설정해 주는 것 (예: 하네스 파이프라인)

> **보충**: PRD는 Product Requirements Document(제품 요구사항 문서)의 약자예요. "하네스(harness)"는 린트·테스트·빌드 같은 검증 도구가 자동으로 돌아가서, 모델이 사람 개입 없이 스스로 결과를 확인하고 고칠 수 있게 만든 실행 환경을 말합니다. 즉 사람의 일이 "코드 검토"에서 "검증 자동화 설계"로 옮겨가는 흐름이에요.

## 4. 바이브 코딩, 어디에 쓰나?

1. **빠른 프로토타이핑** — 프로토타입의 목적은 검증이다. 이 개념이 기술적으로 구현 가능한지, 사용자가 인터페이스를 이해하는지 알아보는 단계이므로 속도가 중요
2. **MVP** — 시장 반응 테스트용 최소 제품. 검증할 질문: 핵심 기능이 동작하는가? 사용자가 돈을 낼 의향이 있는가? 이 비즈니스가 성립하는가? 첫 사용자 10명을 확보할 수 있는가?
3. **내부 도구** — 완성도보다 기능이 중요하다 (대시보드, 슬랙봇, 데이터 마이그레이션 도구 등)
4. **학습·탐색** — 새로운 기술을 배우거나 기능을 빠르게 탐색하고 싶을 때

> **보충**: MVP는 Minimum Viable Product(최소 기능 제품)의 약자로, "완성품"이 아니라 가설을 검증할 수 있는 최소한의 제품을 뜻해요. 네 가지 상황의 공통점은 모두 "코드 자체"가 아니라 "빠른 답"이 목적이라는 점입니다.
